# SSS Behavior Contract v1 — «мозг» без потери доверия

**Назначение:** нормативный слой **поведения системы** (не визуала): трение, стабильность вывода, единый голос, границы того, что продукт **никогда** не делает по умолчанию.

## 0. Три золотых закона SSS

1. **Stability > Intelligence** — лучше стабильный простой вывод, чем «умный», но постоянно меняющийся.
2. **Soft influence > Hard control** — UI направляет, но не запрещает обычные действия пользователя.
3. **One voice > Many opinions** — пользователь видит **один** итоговый нарратив и **одну** доминирующую линию действий (см. DESIGN_SYSTEM §1).

## 1. Friction UX (анти-импульс)

### 1.1 Что запрещено по умолчанию

- **Механическая блокировка** CTA (таймеры «нельзя нажать N секунд») для обычных финансовых действий — ломает доверие и ощущение контроля.
- Любое трение без объяснения, зачем оно пользователю.

### 1.2 Норматив: мягкое замедление намерения (soft friction)

При детектируемом импульсе (по правилам продукта, не обязательно в v1 в UI):

```text
сигнал → лёгкий tone shift
      → CTA визуально менее «агрессивна» (контраст/насыщенность в рамках темы)
      → микротекст-подсказка
      → кнопка остаётся активной сразу
```

Психологическое влияние, а не запрет.

### 1.3 Паттерны

| Паттерн | Суть |
|---------|------|
| **Soft friction (safe mode)** | Без блокировки: лёгкий fade/intensity CTA (−10–20% визуального акцента), короткий hint, уместный haptic по нажатию |
| **Opt-in Protection Mode** | Включает пользователь; есть объяснение и выключение; единственно честный путь к «сильнее вмешиваться» |
| **Hard block** | Только для fraud, критических ошибок, явно опасных действий — не как default для трат/переводов в быту |

## 2. Stable Truth Layer (стабильность «мнения» системы)

### 2.1 Принцип

```text
Insight must not change faster than human perception of change
```

Система не должна ощущаться как **тревожный коуч**, который постоянно передумывает.

### 2.2 Правила

| Правило | Смысл |
|---------|--------|
| **Persistence window** | Минимальное время удержания показанного инсайта перед сменой (в коде уже есть ориентир 2–3 с для hero) — защита от дёрганья |
| **Hysteresis** | Переходы между уровнями риска **асимметричны**: из худшего состояния назад — только при **достаточно сильном** и устойчивом сигнале (например DANGER → SAFE строже, чем SAFE → WATCH) |
| **Confidence smoothing** | Нет мгновенных скачков уверенности/приоритета; сдвиги **постепенные** там, где это влияет на пользовательский текст |
| **One narrative per session** | В рамках сессии пользователь должен удерживать ощущение: «сейчас у меня **такая** финансовая ситуация», а не «приложение снова переобулось» |

### 2.3 Связь с кодом (ориентиры для ревью)

- Окна стабильности и раскрытия инсайта на Home — в виджете hero и связанных провайдерах.
- **Гистерезис velocity → tier:** `VelocityThresholds` + `nextVelocityTierFromHysteresis` в `packages/features_analytics/lib/src/domain/velocity_tier_hysteresis.dart`; память предыдущего tier — `homePublishedFinancialTierProvider` в `home_decision_engine_provider.dart`.
- Дальше: гистерезис для прочих сигналов (доля расхода/прогноз) — по необходимости; новые пороги — с отсылкой к §2.

## 3. Single Voice System (multi-agent только «под капотом»)

### 3.1 Запрещено в UX

- Несколько «голосов» или спорящих агентов в одном экране Decision Mode.
- Визуализация внутренних разногласий моделей как отдельных персонажей/блоков.

### 3.2 Нормативная архитектура

```text
много сигналов (данные, бюджеты, поведение)
  → один decision / rank / conflict-resolution слой
  → одно решение для UI
  → один insight (и иерархия смысл → деталь → метаданные)
  → один primary CTA (если не оговорено иначе в DESIGN_SYSTEM)
```

Любой «multi-agent» — **внутри** scoring/ranking/разрешения конфликтов; пользователь видит **один** итог.

---

## 4. Что система не делает (без явного исключения в PR)

- Не блокирует обычные действия таймерами «для пользы».
- Не меняет главный нарратив «в ноль» на каждом тикe данных без persistence / гистерезиса.
- Не показывает несколько равнозначных «мнений» о состоянии финансов на одном экране.
- Не подменяет **стабильность** ростом «интеллекта» (новые сигналы обязаны проходить через conflict layer и smoothing).

---

## 5. Баланс продуктовой идентичности

Цель — **доверяемый финансовый интерфейс**, а не демонстрация «умного AI» ценой постоянной смены выводов.

При конфликте приоритетов в разработке: сначала §0 этого документа, затем детали UI из DESIGN_SYSTEM и SSS_UI_SYSTEM_V2.
