# SSS UI Design System Blueprint

**Financial Intelligence UI** — контракт дизайн-системы уровня продуктов в духе Monzo / Revolut: не «набор красивых правил», а согласованный язык интерфейса для принятия финансовых решений.

Связанные документы: [ARCHITECTURE.md](ARCHITECTURE.md) (пакеты и границы), реализация примитивов — пакет `packages/ui_components`.

## 1. UI Core Principle (закон системы)

```text
Every screen shows:
1 decision / 1 insight / 1 action
```

**Смысл:** один экран — один доминирующий смысл и один главный призыв к действию (остальное вторично или спрятано в деталь). Это снижает когнитивную нагрузку и конкурирующие акценты.

**Интерпретация:** «один insight» не запрещает второстепенные подсказки, но запрещает два равнозначных «героя» без явной иерархии.

**SSS UX Polish (код):** контракт `UxDecisionView` в `shared_models` + `UxDecisionMapper` в `app` — в UI на главной попадают только **смысл → контекст → подсказка к действию**; сырые метрики движка не показываются без явного «подробнее». См. [ARCHITECTURE.md](ARCHITECTURE.md) §10.

## 2. UI Modes (режимы плотности и задачи)

Один визуальный язык (Manrope, скругления, `colorScheme`, motion из `AppMotion`), **четыре режима** по задаче и плотности.

### 2.1 Decision Mode — главная (`Home`)

**Цель:** быстро понять состояние.

**Структура (логическая):**

```text
BalanceCard или WalletHeroCard (hero, Surface 2)
→ State (safe / caution / danger)
→ один Insight
→ один Forecast / runway (если уместен)
→ одно главное действие (PrimaryActionButton / FAB)
```

**Правила:** максимум воздуха, минимум конкурирующих блоков смысла.

**Ограничения для этого режима:** не превращать первый экран в длинный список «ещё смыслов»; список последних операций — вторичный блок ниже горизонта фокуса.

### 2.2 Action Mode — импорт, review, массовые решения

**Цель:** быстро принять решения.

**Структура:**

```text
SectionHeader / группа (кластер)
→ компактные строки
→ индикатор уверенности (confidence)
→ массовые действия (bulk)
```

**Правила:** высокая плотность, мало текста, много интерактива; без «эмоциональных» декоративных карточек там, где нужна скорость.

### 2.3 Analysis Mode — аналитика

**Цель:** паттерны и сравнения.

**Правила:** данные и графики важнее прозы; сравнения (дельты, периоды) рядом с визуализацией; текст — подписи и короткие инсайты.

**Структура:**

```text
Chart / визуализация
→ сравнение (delta)
→ компактный insight (chip / подпись)
```

### 2.4 Configuration Mode — настройки

**Цель:** предсказуемость и стабильность.

**Правила:** классический Material-список, `SettingsTile` где уместно; без агрессивных motion-экспериментов; опора на `Theme.of(context)`.

## 3. Surface System (три уровня)

Строго **три** уровня поверхности. Смешение уровней без причины разрушает целостность.

| Уровень | Назначение | Примеры в продукте |
|--------|------------|-------------------|
| **Surface 0** | Фон, базовый контейнер экрана | `Scaffold` / фон темы, базовые градиенты под hero |
| **Surface 1** | Списки, нейтральные карточки, строки | `GlassCard`, строки настроек, вторичные карточки |
| **Surface 2** | Фокус, решение, предупреждение, hero | `BalanceCard`, блок Decision Engine на главной, критичные элементы review |

**Правило:** не вводить «Surface 3» и произвольные полупрозрачные контейнеры в фичах без добавления примитива в `ui_components`.

---

## 4. Typography System

**Норма:** для большинства экрана достаточно **четырёх ролей** текста (маппинг на `Theme.of(context).textTheme`):

| Роль | Назначение | Ориентир в Material 3 |
|------|------------|------------------------|
| **Display** | Главный инсайт / крупный акцент | `displaySmall` / `headlineMedium` (по контексту) |
| **Title** | Заголовки секций | `titleLarge` / `titleMedium` |
| **Body** | Основной текст, пояснения | `bodyLarge` / `bodyMedium` |
| **Caption** | Метаданные, время, confidence, вторичное | `bodySmall` / `labelSmall` |

**Запрещено:** случайные веса и размеры в фичах «с нуля»; кастомные `TextStyle` вне темы, если это не исключение с обоснованием.

**Шрифт:** `GoogleFonts.manrope` задаётся в теме приложения (`AppTheme`), не переопределять семейство в фичах.

## 5. Color System (семантика)

**Принцип:** цвет передаёт **смысл**, не украшение.

| Смысл | Семантика | Ориентир |
|--------|-----------|----------|
| Зелёный | Безопасно, хороший тренд, доход | `colorScheme.primary` / `Colors.green` в контексте дохода — согласовать с темой |
| Янтарь / жёлтый | Внимание, неопределённость | `colorScheme.tertiary` / предупреждающие оттенки темы |
| Красный | Риск, перерасход, ошибка | `colorScheme.error` |
| Синий / primary | Нейтральная информация, бренд | `colorScheme.primary` |
| Серый / onSurface с alpha | Неактивно, вторичный фон, метаданные | `onSurface` с прозрачностью, `outline` |

**Правило:** один и тот же смысл — **один** цветовой паттерн на всех экранах (например перерасход всегда через error-семью, а не то красный, то оранжевый).

**Запрещено:** хардкод Hex вне темы для повторяющихся состояний; новые «фирменные» цвета только через `ColorScheme` / расширение темы.

## 6. Motion Language

**Характер (personality):** спокойная предсказуемая система — *calm predictive system*.

**Источник истины:** `packages/ui_components/lib/src/theme/motion_tokens.dart` — класс **`AppMotion`**.

**Правила:**

- Не использовать случайный bounce, «игровые» анимации, хаотичные кривые.
- Длительности и кривые — из `AppMotion` (`fast`, `standard`, `screen`, `staggerInterval`, `curve` / `curveReverse`).

**Паттерны:**

| Событие | Паттерн |
|---------|---------|
| Enter | `fade` + лёгкий `slideY` / `slideX` |
| Update | плавная смена состояния, без резких скачков макета |
| Смена «решения» | лёгкий сдвиг акцента (цвет/контраст), без театра |

**Инструмент:** `flutter_animate` — допустим, если параметры согласованы с `AppMotion`.

## 7. Component Grammar

### 7.1 Правило

Интерфейс собирается из **смысловых блоков** библиотеки. Если примитива нет — сначала расширяется `ui_components`, а не копируется локально в фиче.

### 7.2 Каталог компонентов

| Компонент | Роль | Статус в коде |
|-----------|------|----------------|
| `GlassCard` | Surface 1, нейтральный контейнер инсайта | **Есть** |
| `BalanceCard` | Surface 2, hero баланса | **Есть** |
| `WalletHeroCard` | Surface 2, hero главной (градиент, инсайт, метрики) | **Есть** |
| `OutlinedCircleIconButton` | Круглая иконка в обводке (шапка, toolbars) | **Есть** |
| `EnhancedExpenseCard` | Строка/карточка операции с градиентом | **Есть** |
| `QuickActions` | Быстрые действия главной | **Есть** |
| `SettingsTile` | Configuration Mode | **Есть** |
| `EmptyState` | Пустые состояния, FTUE | **Есть** |
| `SkeletonList` / шиммер | Загрузка | **Есть** |
| `PrimaryScaffold` | Каркас экрана с `AppBar` | **Есть** |
| `PrimaryActionButton` | Главная CTA (по умолч. 56pt, опционально `height`, radius 20, haptic) | **Есть** |
| `SectionHeader` | Якорь секции (`standard` / `mutedLabel` + trailing) | **Есть** |
| `CompactRow` | Плотная строка Action Mode (акцент, title, subtitle, `belowSubtitle`, trailing) | **Есть** |
| `ConfidencePalette` / `ConfidenceDot` | Цвет/точка по уверенности 0..1 (импорт, review) | **Есть** |
| `InsightChip` | Сжатый инсайт + тон (`neutral` … `negative`) | **Есть** |
| `RiskBadge` | Бейдж `RiskLevel` safe / caution / danger | **Есть** |
| `DecisionTile` | Единица выбора в decision/review | **Цель** — вынести из фич при повторении |

Компоненты со статусом **Цель** не блокируют работу, но новые экраны должны проектироваться так, чтобы дублирующая разметка постепенно заменялась ими.

## 9. Information Hierarchy

На экране информация читается в трёх уровнях (сверху вниз по важности):

```text
1. Meaning   — что важно прямо сейчас
2. Detail    — почему это важно
3. Metadata  — цифры, confidence, время
```

**Типичная ошибка:** показывать сначала уровень 3 (сырые числа) без уровня 1.

## 10. Visual Consistency Checklist (перед merge UI)

Использовать для PR, затрагивающих целый экран или новый поток.

### 10.1 Structure

- [ ] Один главный смысл на экране явно выделен
- [ ] Нет двух равнозначных конкурирующих акцентов без иерархии

### 10.2 Surface

- [ ] Используются только Surface 0 / 1 / 2 по смыслу
- [ ] Нет случайных вложенных контейнеров «просто с тенью»

### 10.3 Typography

- [ ] Роли текста укладываются в четыре типа (§4)
- [ ] Нет произвольных `TextStyle` в фиче без причины

### 10.4 Color

- [ ] Семантика цвета согласована с §5
- [ ] Нет декоративных цветов вместо смысловых

### 10.5 Motion

- [ ] Длительности и кривые из `AppMotion` (или эквивалентно согласованы)
- [ ] Нет случайных анимаций ради эффекта

### 10.6 Components

- [ ] По возможности используются примитивы из `ui_components`
- [ ] Нет клонов компонентов в фиче без задачи на вынос в библиотеку

### 10.7 Cognitive load

- [ ] Смысл экрана читается за 2–3 секунды
- [ ] Нет визуального шума (лишние рамки, иконки, текст)

### 10.8 Decision clarity

- [ ] Понятно, что делать дальше
- [ ] Одно главное действие очевидно (или явно объяснено, если их несколько осознанно)

---

## 11. Финальная формулировка

SSS UI — это не про «красоту ради красоты», а про ощущение, что **интерфейс поддерживает решение**: состояние ясно, действие очевидно, детали не мешают главному.
