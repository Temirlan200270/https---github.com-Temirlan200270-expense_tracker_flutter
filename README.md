# Expense Tracker — Модульный Flutter приложение для учёта расходов

Современное приложение для отслеживания личных финансов с поддержкой бюджетов, долгов, аналитики и импорта данных.

Цель развития продукта — выйти за рамки «ещё одного трекера»: **система принятия финансовых решений**, которая понимает поведение, предупреждает и помогает менять привычки (см. раздел ниже).

## 🧠 Продуктовое видение: SSS-уровень (Financial Brain)

**Обычный трекер** фиксирует траты. **SSS-уровень** — цикл **понимание → анализ → предсказание → влияние** на поведение.

| Обычный трекер | Целевой продукт |
|----------------|-----------------|
| Считает | Думает (паттерны, контекст) |
| Показывает цифры | Объясняет и подсказывает |
| Пассивный | Активный (инсайты, анти-импульс) |
| Сухая аналитика | Вовлечение без кринжа (геймификация опционально) |

### Ядро: Financial Brain (пайплайн)

Логика считается **детерминированно** на устройстве; **AI** — для объяснений, тона и диалога, а не как единственный источник истины по категориям и лимитам.

```
User Input
    → Categorization Engine (правила + история + шаблоны)
    → Behavior Engine (паттерны: день недели, категории, импульсивность)
    → Prediction Engine (прогноз остатка, «не хватит на X»)
    → Advice Engine (формулировки; при необходимости — AI поверх фактов)
    → UI (Flutter)
```

**Behavior Engine (код, `features_analytics/lib/src/behavior_engine/`):**

- `ExpectedSpendingCalculator` / `SpendingVelocityAnalyzer` / `SpendingDeviationDetector` (пороги 1.3 / 0.7).
- `CategoryContributionAnalyzer` — вклад по категориям «сегодня до сейчас» vs норма; топ-«виновник» для строки **«Основная причина…»** при общем перерасходе.
- `InsightConfidenceScorer` — high / medium / low от объёма baseline и CV; разные формулировки лида.
- Главная: `homeDecisionEngineProvider` — предиктивный статус (баланс + прогноз + velocity), один инсайт: сначала общий ≥1.3, иначе fallback по категории (≥1.15), иначе «еда» при ≥1.3.

### Поведенческий слой (что считать)

- Паттерны по **дням недели**, **категориям**, **времени суток**
- Оценка **импульсивных** трат (эвристики + пользовательские метки, при желании — настроение)
- **Контекст**: календарь, привычки, сравнение с «своей нормой», а не только с прошлым месяцем

### Примеры ценности (не «потратил 5000», а смысл)

- «На еду по вторникам у тебя **+32%** к обычной неделе»
- «При текущем темпе **через ~N дней** закончится запас до зарплаты»
- «Бюджет «Развлечения» уже **+80%** к лимиту — обычно в такие дни ты потом жалеешь»

### Умные фичи (roadmap по приоритету)

**Фаза 1 — фундамент (сначала это):**

1. **Авто-категоризация** — название, история, правила (минимум ручного ввода)
2. **Базовые инсайты** — доли, аномалии по дням/категориям, простые текстовые выводы
3. **Простое предсказание** — линейный/скользящий прогноз расхода до даты (зарплата, конец месяца)

**Фаза 2 — влияние на поведение:**

- **«Ты сейчас ошибаешься»** — пороги бюджета, повторяющиеся паттерны «жалобы» (если введёте учёт)
- **Anti-impulse** — пауза перед записью/подтверждением, явный вопрос «импульс?»
- **Экран аналитики как «истории»** — не таблицы и не сухие графики в приоритете; главный экран: баланс, **финансовое состояние**, 1–2 ключевых инсайта

**Фаза 3 — углубление:**

- **AI** — только объяснения и рекомендации **поверх** уже посчитанных метрик
- **Финансовая «энергия» / дисциплина** — лёгкая геймификация (уровень дисциплины, EXP за бюджет, штрафы за импульс), в духе мотивации без перегруза UI

### Риски (где «облажаться»)

- Слишком много функций сразу, перегруз UI, раздувание AI, слабый UX при вводе траты (должно оставаться **1–2 тапа** + автозаполнение)

### Критерий «готово к SSS»

Продукт на SSS-уровне, когда одновременно работают **Behavior Engine + Prediction + Influence** (хотя бы в минимальном виде), а не когда «красиво нарисовали графики».

### План SSS vNext: от «сильного проекта» к выдающемуся продукту

**Коротко:** закрыто примерно **80–85%** SSS-ядра (сравнение с нормой, главный экран, категоризация rules+history, базовый confidence лидов через `InsightConfidenceScorer`). Дальше — **не дублируем очевидное**; только то, что реально **не учтено или недодумано** в типичном roadmap и отделяет «очень сильный проект» от **реально выдающегося** продукта.

#### 1. Инерция поведения (тренд)

- **Сейчас:** в фокусе `deviation = текущее vs среднее` — система в основном **реактивная**.
- **Пробел:** не видно **тренда изменения поведения** (динамика во времени).
- **Пример:** вчера +10%, позавчера +5%, сегодня +30% — недостаточно одной фразы «+30% перерасход»; сильнее: **«ты ускоряешься»** (нарастание отклонения).
- **Направление:**

```text
trend = slope(last N days deviation)
```

Смысл: из «реагирует на срез» → в **«предсказывает направление поведения»**.

#### 2. Событийность (event detection)

- **Сейчас:** все дни формально равны; нет отдельного слоя **событий**.
- **Пробел:** не различаются зарплата, выходные, редкие крупные покупки — они **ломают норму** и дают **ложные инсайты**.
- **Направление — Event Detection Layer**, эвристика уровня:

```text
если расход >> обычного × 3 → это событие, не новый паттерн
```

События учитывать в статистике/базлайне отдельно от «типичного дня».

#### 3. Окно адаптации (adaptation)

- **Сейчас:** «норма» опирается на историю целиком.
- **Проблема:** смена режима (начал экономить / тратить больше) долго выглядит как **аномалия**.
- **Направление — decay / weighting**, недавнее важнее старого, например:

```text
recent_data_weight = 0.7
old_data_weight    = 0.3
```

(конкретные коэффициенты — подобрать по данным и UX.)

#### 4. Confidence инсайтов (углубление)

- **Сейчас:** deviation считается; **базовый** confidence инсайтов уже есть (`InsightConfidenceScorer` → high/medium/low для лидов).
- **Пробел:** не сведено в одну явную картину влияние **мало данных**, **шума**, **нестабильных паттернов** на все типы выводов.
- **Направление — Insight Confidence Score** в духе:

```text
confidence ≈ sample_size_factor × variance_factor × consistency_factor
```

**UX:** высокий confidence → уверенный текст; низкий → мягкий (**меньше ощущения рандома**).

#### 5. Cost of mistake (вес ошибки)

- **Сейчас:** ошибки/отклонения **морально равны** независимо от масштаба.
- **Проблема:** отклонение на **500 ₸** и на **50 000 ₸** для пользователя не равнозначны.
- **Направление:** вес влияния, напр. `impact ∝ deviation × сумма` (или нормированная сумма) → отдельные формулировки: **«отклонение критично из‑за крупной суммы»**.

#### 6. Память сожалений (Regret Memory)

- В видении уже есть «обычно жалеешь»; **отдельного продуктового контура** пока нет.
- **Простой вариант:** после траты — «жалеешь?» да/нет → при **похожей** трате мягкое предупреждение (личный опыт, не абстрактная статистика).

#### 7. Decision layer (что делать)

- **Сейчас:** сильная зона — **«что происходит»**.
- **Пробел:** слабее **«что делать»** в явном виде.
- **Направление — Actionable suggestions:** «снизь траты на X сегодня», «подожди до завтра», «лимит уже достигнут» — перевод инсайта в **действие**.

#### 8. Самый большой риск

Сделать всё перечисленное **сразу** = высокая вероятность **переусложнения** и шума в UI. **Не делать всё параллельно.**

#### Реальный приоритет внедрения

1. **Trend (инерция)** — максимальный сдвиг от реактивности к пониманию динамики.
2. **Adaptation (decay)** — норма «едет» вместе с пользователем.
3. **Confidence (расширение)** — единая логика и тон формулировок.

**Позже итерациями:** события, cost of mistake, regret memory, actionable layer.

#### Чеклист «что ещё не закрыто»

- Тренды (динамика отклонения)
- События (редкие/крупные выбросы)
- Адаптация baseline (вес недавнего)
- Углублённая уверенность инсайтов
- Вес ошибки по сумме / impact
- Память сожалений
- Явные действия поверх инсайтов

#### Уже сделано (для контекста)

- Система **понимает** и **сравнивает** (Behavior Engine, главный экран, категоризация).
- **Следующий скачок SSS:** система уверенно описывает **динамику поведения во времени**, а не только мгновенный срез vs среднее.

**Возможные шаги в коде (когда возьмёшься):** `Behavior Engine v2` — пакетно trend + decay в `features_analytics` / `behavior_engine.dart`, расширение confidence рядом с `InsightConfidenceScorer`; остальное отдельными фичами.

## 📋 План расширения продукта (Supabase, UI, голос, пуши, виджеты)

Ниже — целевой бэклог **рядом** с SSS vNext: инфраструктура, ввод, каналы уведомлений и поверхность «вне приложения». Оценки сроков ориентировочные; реальный продуктовый уровень тянет **backend + правила + слой данных**.

### 1. Supabase (и облако)

- Подключить **Supabase** как целевой бэкенд: синхронизация данных, опционально auth и хранение настроек/флагов.
- В коде — развитие контракта `SyncClient` / `sync_api` и подмена мока в `bootstrap` (аналогично примеру с Firebase в этом README): **схема таблиц, конфликты, миграции**.
- Пуши и виджеты часто опираются на **стабильный снимок данных в облаке** — закладывать Supabase **до** или параллельно с «умными» пушами.

### 2. Добить UI

- Завершить выравнивание по **[DESIGN_SYSTEM.md](DESIGN_SYSTEM.md)** и **[docs/SSS_UI_SYSTEM_V2.md](docs/SSS_UI_SYSTEM_V2.md)** (чеклист §10, при необходимости `dart tool/sss_ui_audit.dart --strict`).
- Цель: **заморозка визуального слоя** без параллельного наращивания логики поведения (см. разделение «форма / смысл» в документах).

### 3. Голосовой ввод + автоматическая категоризация

- Ввод траты **голосом** (распознавание речи на устройстве или через сервис).
- После распознавания — тот же пайплайн, что и для текста: **Categorization Engine** (правила + история + шаблоны), при необходимости уточнение в UI перед сохранением.
- Риски: качество ASR, локали RU/EN, приватность (on-device vs облако).

### 4. Push-уведомления (Firebase Cloud Messaging / канал через Supabase)

| Уровень | Содержание | Ориентир усилий |
|--------|------------|-----------------|
| **Базовый** | Напоминание о бюджете; ежедневный чек-ин; локальные пуши или FCM **без** сложной логики на сервере | порядка **1–2 дней** на интеграцию канала и простые триггеры |
| **Продуктовый** | Триггеры из данных (перерасход, паттерны); поведенческие уведомления; персонализация времени; анти-спам / throttling; согласованность между устройствами | **backend + rules engine + data layer**; отдельный этап проектирования |

### 5. Виджеты на главном экране (Android / iOS)

| Уровень | Содержание | Ориентир усилий |
|--------|------------|-----------------|
| **Проще** | Баланс; последние расходы; 1–2 кнопки (например «добавить трату») | порядка **2–4 дней** на первую версию под одну платформу |
| **Сложнее** | Фоновое обновление; lifecycle App Widget (Android); **WidgetKit** + SwiftUI (iOS); синхронизация с облаком; **кеш снимка** данных; fallback, когда приложение закрыто | **нативный слой + система снимков данных**; существенно больше, чем Flutter-only |

**Зависимость:** стабильный **кеш/снимок** для виджета (локальный или из Supabase) критичен раньше, чем «умные» пуши на все случаи.

### Рекомендуемый порядок внедрения (без параллельного хаоса)

1. **Supabase** — схема синка и рабочий поток данных.
2. **UI freeze / добивка** по дизайн-системе.
3. **Голос + категоризация** — внутри уже существующего движка правил.
4. **Пуши:** сначала базовый уровень → затем продуктовый слой по мере готовности правил и данных.
5. **Виджеты:** простой виджет → кеш/снимок → усложнение и второй экран ОС.

### Замена мокового SyncClient на Firebase/Supabase

```dart
// packages/sync_api/lib/src/firebase_sync_client.dart
class FirebaseSyncClient implements SyncClient {
  FirebaseSyncClient(this._firestore);

  final FirebaseFirestore _firestore;

  @override
  Future<void> pushExpenses(List<Expense> expenses) async {
    final batch = _firestore.batch();
    for (final expense in expenses) {
      final doc = _firestore.collection('expenses').doc(expense.id);
      batch.set(doc, expense.toJson(), SetOptions(merge: true));
    }
    await batch.commit();
  }

  @override
  Future<List<Expense>> pullExpenses({DateTime? since}) async {
    var query = _firestore.collection('expenses');
    if (since != null) {
      query = query.where('updatedAt', isGreaterThan: since.toIso8601String());
    }
    final snapshot = await query.get();
    return snapshot.docs.map((doc) => Expense.fromJson(doc.data())).toList();
  }
}
```

Дальше просто переопредели `syncClientProvider` в `packages/app/lib/bootstrap.dart` на `FirebaseSyncClient` или адаптер Supabase.

## 🔄 Замена мокового SyncClient на Firebase/Supabase

Приложение готово к интеграции с облачной синхронизацией. Пример для Firebase:

```dart
// packages/sync_api/lib/src/firebase_sync_client.dart
class FirebaseSyncClient implements SyncClient {
  FirebaseSyncClient(this._firestore);

  final FirebaseFirestore _firestore;

  @override
  Future<void> pushExpenses(List<Expense> expenses) async {
    final batch = _firestore.batch();
    for (final expense in expenses) {
      final doc = _firestore.collection('expenses').doc(expense.id);
      batch.set(doc, expense.toJson(), SetOptions(merge: true));
    }
    await batch.commit();
  }

  @override
  Future<List<Expense>> pullExpenses({DateTime? since}) async {
    var query = _firestore.collection('expenses');
    if (since != null) {
      query = query.where('updatedAt', isGreaterThan: since.toIso8601String());
    }
    final snapshot = await query.get();
    return snapshot.docs.map((doc) => Expense.fromJson(doc.data())).toList();
  }
}
```

Переопределите `syncClientProvider` в `packages/app/lib/bootstrap.dart`:

```dart
syncClientProvider.overrideWithValue(FirebaseSyncClient(FirebaseFirestore.instance)),
```
